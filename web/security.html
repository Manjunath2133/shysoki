Security Specification

Display Protection & Privacy Policy

A technical whitepaper explaining the shielding mechanisms, data collection boundaries, and local encryption models built into Shyoski.

1. Display Capture Exclusion Policy

Shyoski prevents any operating system-level recording or snapshotting of its visual workspace. We implement native operating system hooks to flag the application window context:

  • macOS APIs: We configure the underlying Cocoa window representation to enforce `sharingType = NSWindowSharingNone` and register the browser frame inside the Apple Quartz Window Server framework with `excludesFromCapture = true`.
  • Windows APIs: On Windows platforms, we trigger the Desktop Window Manager (DWM) hook: `SetWindowDisplayAffinity(hwnd, WDA_EXCLUDEFROMCAPTURE)`.

Result: Any proctoring browser extension, screen sharing tool (Zoom, Teams, Discord), or background recording utility (OBS, Loom) will record a blank, transparent window instead of our overlay interface, keeping your workspace private.

2. Audio Data and Stream Isolation

Our transcription engines work completely locally on your hardware. We capture microphone loops and loopback devices using isolated thread pipes:

  • Audio bytes are kept in volatile heap buffers for a maximum of 400 milliseconds to translate or match voice prints.
  • No Local Storage: Shyoski never commits audio recordings, transcription databases, or session voice records to the local filesystem.
  • When the listening toggle is turned off, all memory pools allocated to stream audio are wiped immediately.

3. Local Cryptographic Cache

To support offline execution checks and billing sync integrity, the application caches a encrypted session token in `billing_config.json`. We leverage local hardware keystores to seal the file contents:

  • macOS: Encryption keys are sealed inside the secure Apple Keychain system.
  • Windows: We use the DPAPI (Data Protection API) where keys are tied to the local logged-in Windows user account password.

4. Concurrency & Anti-Tampering Checks

To prevent concurrent license bypasses, the server checks unique system attributes (fingerprints consisting of CPU IDs and system serial numbers) hashed with SHA-256. If a timestamp drifting attempt (manipulating the system clock) is registered, the application locks keyboard control in Ghost Mode until synced with the server.
